# R1 Review — Percy Liang
**Paper**: B.16 ConvergenceSweep: T=600 Statutory Seed Formula
**Score**: 3.0/4
**Verdict**: Minor Revision

## Summary
B.16 introduces ConvergenceSweep, establishes T=600 as the statutory stopping criterion via 50-state empirical sweep data, and provides a complete implementation specification including SHA-256 seed derivation, pseudocode, and a tamper-evident audit log design. From a reproducibility standpoint, this is the most technically rigorous paper in the B-series: Algorithm 1 is fully specified, the determinism proof identifies both sources of non-determinism (seed sequence and floating-point non-associativity), and the JSONL deposition log with SHA-256 chaining is an exemplary audit trail. Three specification gaps need resolution before publication.

## Strengths
- Algorithm 1 is specified at the right level of abstraction: the pseudocode starts by computing s_0, initialises EC* from M(G, k, s_0), and uses a strict less-than comparison (e_j < EC*) to ensure the first minimum is retained in case of ties. This is production-quality algorithm specification that prevents non-determinism in the comparison step.
- The implementation audit trail references specific crate (redist-core), specific file (seed_compositor.rs), and specific enum variants (Fixed, ConvergenceSweep, Ensemble). The FIPS 180-4 compliance citation for the sha2 crate and the Cargo.lock pinning mechanism together constitute the most complete reproducibility infrastructure in any redistricting paper to date.
- The deposition log design (Section 5.3) closes the "what was computed" problem: recording EC_norm at every seed tried, the winning seed index j*, and a SHA-256 hash of the output map in a chained JSONL format allows post-hoc verification of the entire sweep. The chain structure means any modification to a historical entry invalidates all subsequent hashes.

## Concerns
1. **METIS determinism requirement is underspecified for parallel implementations.** Proposition 1 concludes that ConvergenceSweep is deterministic "regardless of hardware, operating system, or parallelisation strategy applied to independent METIS calls." This is too strong: METIS's OpenMP-parallel variants use a non-deterministic work-stealing scheduler, so two runs with the same seed but different thread-scheduling outcomes may produce different local optima. The proposition must be qualified to specify whether the statutory run uses single-threaded METIS (METIS_OPTION_NTHREADS=1, in which case the guarantee holds) or multi-threaded METIS with a fixed thread count (in which case the thread count must be part of the reproducibility specification). The current wording would allow a verifier to run multi-threaded METIS and obtain a different plan, then claim the statutory run is non-reproducible.
2. **Table 5 runtime figures are not clearly labelled as measured or estimated.** The caption says "estimated ... based on observed METIS call times" but the j_stop column says "empirical seeds tried," implying the sweeps were actually run. If sweeps were run (Georgia: 1,111 seeds at 0.31s each), the times are measured and should be presented as such. If times are computed as j_stop * T(M) from a per-seed timing measurement, T(M) should be shown as a measured quantity and the total as a derived quantity. The distinction matters: a measured 5.7 minutes for Georgia is a reproducibility claim; an estimated 5.7 minutes is a projection.
3. **Cargo.lock is referenced but not made publicly accessible.** The paper references Cargo.lock for METIS source hash pinning and sha2 crate version pinning, and Section 5.1 states the statutory run is pinned to a specific source hash. But no URL is provided to the repository or the specific commit used to generate Table 1. For the "certifiably optimal" claim to be independently verifiable, the exact Cargo.lock used for the 50-state B.7 sweep must be publicly accessible. Without this, the reproducibility claim cannot be independently confirmed.
4. **The interaction between ConvergenceSweep and zero-seed-variance structures is unaddressed.** The statutory invocation in Section 5.2 specifies --structure prime-factor (ApportionRegions) with --search convergence. B.11 established that ApportionRegions has zero seed variance — all seeds produce the same plan. If AR has zero seed variance, ConvergenceSweep would terminate at T=1 because the second seed never improves on the first. Does the statutory command actually run 600+ METIS calls for an AR run, or does it terminate immediately? This interaction needs clarification to avoid statutory runs that complete in one seed and mislead users about the sweep behavior.

## Required Changes
- **P1**: Qualify Proposition 1 to specify that the determinism guarantee requires single-threaded METIS (METIS_OPTION_NTHREADS=1). Add a note that the statutory build specifies this flag explicitly.
- **P1**: Clarify Table 5 as reporting measured times (if sweeps were run) or as projections from measured per-seed T(M) values (if not). Add a footnote specifying which states were directly measured.
- **P2**: Provide a URL to the specific repository commit containing the Cargo.lock used for the 50-state B.7 sweep, or state that it will be provided upon acceptance.
- **P2**: Add a note addressing how ConvergenceSweep behaves when --structure prime-factor is used with zero seed variance; specify whether the sweep runs to T seeds or terminates early.
