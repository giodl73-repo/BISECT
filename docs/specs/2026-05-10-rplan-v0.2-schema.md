## Purpose

RPLAN v0.1 is a useful tract-assignment interchange format. It is not enough
for the generic `rplan-*` crate family because it assumes Census tracts,
GEOID-keyed assignment maps, 1-based district ids, and no canonical plan hash.

RPLAN v0.2 defines the generic plan identity model used by `rplan-core`.
RPLAN v0.1 remains supported as a compatibility input.

## File Model

A `.rplan` file is a plan assignment artifact. It is not required to contain
adjacency, population, legal rules, or audit results.

```json
{
  "rplan_version": "0.2",
  "plan": {
    "schema_version": "district-plan-v1",
    "units": { "...": "..." },
    "assignment": [0, 0, 1, 1],
    "k": 2,
    "display_labels": ["1", "2"],
    "allow_empty_districts": false
  },
  "metadata": { "...": "..." },
  "provenance": { "...": "..." },
  "geometry": null,
  "extensions": {}
}
```

Top-level required fields:

| Field | Type | Description |
|-------|------|-------------|
| `rplan_version` | string | Must be `"0.2"` for this schema. |
| `plan` | object | Canonical district-plan assignment model. |
| `metadata` | object | Human-facing plan description. |
| `provenance` | object | Tool-neutral producer and source identity. |
| `geometry` | object\|null | Optional RFC 7946 district geometry. |
| `extensions` | object | Optional namespaced extension payloads. |

## Plan Object

```json
{
  "schema_version": "district-plan-v1",
  "units": {
    "unit_kind": "tract",
    "state": "WA",
    "year": 2020,
    "canonical_order": "explicit-unit-ids",
    "unit_ids": ["53001000100", "53001000200"],
    "unit_universe_hash": "sha256:..."
  },
  "assignment": [0, 1],
  "k": 2,
  "display_labels": ["1", "2"],
  "allow_empty_districts": false
}
```

### `units`

| Field | Required | Type | Description |
|-------|----------|------|-------------|
| `unit_kind` | yes | string | `block`, `block-group`, `tract`, `county`, `precinct`, or `imported`. |
| `state` | no | string | Jurisdiction code when applicable. |
| `year` | no | integer | Census/election/data vintage when applicable. |
| `canonical_order` | yes | string | `explicit-unit-ids` or `sorted-geoid`. |
| `unit_ids` | yes | array string | Ordered unit ids. |
| `unit_universe_hash` | yes | string | Hash binding this plan to a known unit universe. |

Rules:

- `assignment.length == unit_ids.length`.
- `assignment[i]` is the canonical district id for `unit_ids[i]`.
- `unit_ids` must be unique.
- For `sorted-geoid`, every `unit_id` must be numeric and the array must be
  ascending lexicographically.
- For `explicit-unit-ids`, array order is authoritative and participates in the
  plan hash.

Per-kind unit-id validation:

| `unit_kind` | Validation |
|-------------|------------|
| `tract` | exactly 11 ASCII digits |
| `block-group` | exactly 12 ASCII digits |
| `block` | exactly 15 ASCII digits |
| `county` | exactly 5 ASCII digits |
| `precinct` | non-empty string; should be namespaced when not globally unique |
| `imported` | non-empty string; no Census format assumption |

### `assignment`

RPLAN v0.2 stores canonical internal district ids as integers in `0..k-1`.
Every id in `0..k-1` must appear at least once unless
`plan.allow_empty_districts` is explicitly true. The default is false.

External labels belong in `display_labels`. For v0.1 compatibility,
`rplan-io` maps v0.1 labels `1..k` to v0.2 ids `0..k-1` and sets:

```json
"display_labels": ["1", "2", "..."]
```

Algorithm crates must not perform their own label conversion.

`display_labels` are presentation fields. They do not participate in
`plan_hash`. A file that changes only district display labels represents the
same district assignment with different presentation.

### Unit-Universe Hash

`unit_universe_hash` binds the plan to the unit universe. It is computed over
this canonical projection:

```json
{
  "unit_kind": "tract",
  "state": "WA",
  "year": 2020,
  "canonical_order": "explicit-unit-ids",
  "unit_ids": ["53001000100", "53001000200"],
  "source_id": "tiger-line-2020"
}
```

Rules:

- `source_id` is optional, but when present it participates in the hash.
- absent optional fields and explicit `null` are different values.
- the same canonical JSON rules used for `plan_hash` apply here.
- the hash string is `sha256:<lowercase-hex>`.

For v0.1 compatibility, `rplan-io` computes `unit_universe_hash` from:

```json
{
  "unit_kind": "tract",
  "state": "<metadata.state_code>",
  "year": <metadata.year as integer>,
  "canonical_order": "sorted-geoid",
  "unit_ids": ["..."]
}
```

## Metadata

Metadata is human-facing and should not be used as the only source of audit
truth.

```json
{
  "label": "wa_house_draft1",
  "jurisdiction": "WA",
  "chamber": "state-house",
  "description": "Draft 1",
  "created_at": "2026-05-10T00:00:00Z"
}
```

Required fields:

- `label`
- `jurisdiction`
- `chamber`
- `created_at`

Allowed `chamber` values:

- `congressional`
- `state-house`
- `state-senate`
- `local`
- `custom`

## Provenance

RPLAN v0.2 replaces v0.1's `source_manifest` with tool-neutral provenance.

```json
{
  "producer": {
    "name": "bisect",
    "version": "0.1.0",
    "git_commit": "..."
  },
  "source_hashes": {
    "unit_universe": "sha256:...",
    "pl_94171": "sha256:...",
    "tiger_line": "sha256:...",
    "adjacency": "sha256:..."
  },
  "conversion_lineage": []
}
```

`bisect-cli` may adapt `PlanManifest` into this object, but `rplan-*` crates
must not depend on `PlanManifest`.

## Geometry

When present, `geometry` is a GeoJSON FeatureCollection following RFC 7946:

- WGS84 longitude/latitude coordinates.
- no `crs` member.
- exterior rings counterclockwise, interior rings clockwise.
- each feature has `properties.district_id` using the canonical `0..k-1` id.
- `properties.display_label` may carry a human district label.

Projected or NAD83 geometry must be represented outside the embedded GeoJSON
object, either by an extension payload or by a future context artifact.

## Extensions

Extensions must be namespaced:

```json
{
  "extensions": {
    "org.example.tool": {
      "schema_version": "example-v1",
      "payload": {}
    }
  }
}
```

Extensions must not change the meaning of required RPLAN fields. Readers may
ignore unknown extensions.

## Canonical Hashing

RPLAN v0.2 defines a canonical plan hash over this projection:

```json
{
  "schema_version": "district-plan-v1",
  "units": {
    "unit_kind": "...",
    "state": "...",
    "year": 2020,
    "canonical_order": "...",
    "unit_ids": ["..."],
    "unit_universe_hash": "sha256:..."
  },
  "assignment": [0],
  "k": 1,
  "allow_empty_districts": false
}
```

Canonical JSON rules:

- UTF-8 encoding.
- no insignificant whitespace.
- object keys sorted lexicographically by Unicode code point.
- arrays preserve order.
- integers are base 10 with no leading zeros.
- finite floats use shortest-roundtrip formatting; non-finite floats are
  forbidden in canonical-hash projections.
- absent optional fields and explicit `null` are different values.

The hash string is:

```text
sha256:<lowercase-hex-sha256(canonical-json-bytes)>
```

`display_labels`, `metadata`, `provenance`, `geometry`, and `extensions` are
not part of the canonical plan hash unless a later schema explicitly includes
them in a separate document or presentation hash.

## Format Validity Vs Audit Validity

RPLAN readers must distinguish:

- **format-valid**: the `.rplan` file obeys this schema.
- **context-valid**: the plan binds to the supplied unit universe/context.
- **audit-valid**: the plan passes selected checks under a legal/audit profile.

A `.rplan` file can be format-valid without being auditable for contiguity or
population. `rplan-audit` consumes a plan plus context plus profile and emits an
audit certificate.

## Context Artifact

RPLAN v0.2 reserves `.rctx` for context:

```json
{
  "rctx_version": "0.1",
  "context_hash": "sha256:...",
  "units": {
    "unit_kind": "tract",
    "state": "WA",
    "year": 2020,
    "canonical_order": "explicit-unit-ids",
    "unit_ids": ["53001000100", "53001000200"],
    "unit_universe_hash": "sha256:..."
  },
  "graph": {
    "edge_semantics": "undirected",
    "adjacency": [
      [{"to": 1, "kind": "boundary", "weight": 120.5}],
      [{"to": 0, "kind": "boundary", "weight": 120.5}]
    ]
  },
  "populations": [100, 100],
  "source_hashes": { "...": "..." }
}
```

The `.rctx` schema is owned by `rplan-core`/`rplan-io`, but may evolve
separately from `.rplan`.

Top-level required fields:

| Field | Type | Description |
|-------|------|-------------|
| `rctx_version` | string | Context schema version. Initial value: `"0.1"`. |
| `context_hash` | string | Hash of the canonical context projection. |
| `units` | object | Same `PlanUnitIndex` shape used by `.rplan`. |
| `graph` | object\|null | Optional unit adjacency graph. |
| `populations` | array integer\|null | Optional population vector aligned to `units.unit_ids`. |
| `source_hashes` | object | Source hashes for units, graph, populations, geometry, and other inputs. |

`context_hash` is computed over `units`, `graph`, `populations`, and
`source_hashes` with the same canonical JSON rules as `plan_hash`.

Minimum graph model:

```rust
pub struct UnitGraph {
    pub units: PlanUnitIndex,
    pub adjacency: Vec<Vec<UnitEdge>>,
}

pub struct UnitEdge {
    pub to: u32,
    pub kind: EdgeKind,
    pub weight: Option<f64>,
}
```

Allowed `EdgeKind` values:

- `boundary`
- `point-touch`
- `bridge`
- `ferry`
- `water`
- `custom`

Graph rules:

- On disk, v0.1 `.rctx` adjacency is undirected and symmetric.
- If unit `i` lists an edge to `j`, unit `j` must list the corresponding edge
  to `i`.
- duplicate edges are invalid.
- `to` must be a valid index into `units.unit_ids`.
- `adjacency.length == units.unit_ids.length`.
- `populations.length == units.unit_ids.length` when populations are present.
- weights are optional finite non-negative floats.

The graph is optional for interchange and required for contiguity audit.

## v0.1 Compatibility

`rplan-io` must support v0.1 reading:

- top-level `assignments` map is read as tract units.
- assignment values `1..k` convert to internal `0..k-1`.
- `display_labels` is generated from the original labels.
- `unit_ids` are sorted lexicographically by GEOID.
- `unit_universe_hash` is computed from the sorted unit ids unless an external
  context supplies a stronger source hash.
- v0.1 `source_manifest` is preserved as a namespaced extension, not as a
  `rplan-core` dependency.

`rplan-io` may write v0.1 only for backwards compatibility; new writers should
emit v0.2.

## Acceptance Criteria

- v0.2 example parses into `rplan-core::DistrictPlan`.
- v0.1 fixture converts deterministically to v0.2 internal form.
- canonical plan hash is stable across object-key ordering changes.
- two files with identical plan assignment but different metadata have the same
  plan hash.
- changing `unit_ids`, `assignment`, `k`, or `allow_empty_districts` changes the plan
  hash.
- changing only `display_labels` does not change `plan_hash`.
- `.rplan` format validation does not require graph or population context.
- a 5-node path `.rctx` fixture round-trips through `rplan-io`.
- `rplan-audit` rejects contiguity checks when `.rctx` graph context is missing.
