# Goal: Complete U.20 Plan Audit Integration

**Status:** Active after RPLAN Phase 1  
**Track:** U.20 - Search and Optimization / audit  
**Foundation:** [`2026-05-10-rplan-phase1-checklist.md`](2026-05-10-rplan-phase1-checklist.md)

## Goal

Complete U.20 by turning the RPLAN audit foundation into end-to-end audit
certificates for generated, solver-backed, and selected plans, with lineage,
verification, reporting, and L1/L2 coverage.

## Current Baseline

- RPLAN Phase 1 is complete.
- `rplan-core`, `rplan-io`, `rplan-audit`, and `rplan-cli` are in place.
- `bisect-cli::runner` emits `.rplan`, `.rctx`, and `audit-certificate.json`.
- `PlanManifest`, `.rplan` provenance, `.rctx` source hashes, reports, and
  `bisect verify` are tied together by source-hash and certificate checks.

## Finish Criteria

- [x] U.16 solver-backed plans populate rich `algorithm_lineage`.
- [x] ILP lineage includes solver mode, bounds/gap, fallback, and audit summary
      identity without duplicating reserved certificate fields.
- [x] Future algorithm-family producers can attach lineage through stable
      `rplan-audit` APIs rather than ad hoc manifest fields.
- [ ] Selected Pareto/frontier plans can carry per-plan validity status and, for
      exported selections, full audit certificates.
- [x] `bisect verify` covers solver lineage identity where available.
- [x] JSON/HTML reports surface lineage in the audit trail.
- [ ] L1 coverage verifies a complete generated audit package, including
      source hashes, certificate identity, context hash, and lineage.
- [ ] L2 coverage exercises at least one real runner path that emits and verifies
      the full audit package.

## Next Stage

Start with U.16/ILP lineage integration because it is the first concrete
producer beyond generic runner provenance. Keep new lineage fields in
`algorithm_lineage.extra` and preserve the existing certificate schema boundary.
